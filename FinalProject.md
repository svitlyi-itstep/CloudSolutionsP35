# Створення хмарних рішень з використанням AWS

## ЗАВДАННЯ НА ФІНАЛЬНИЙ ПРОЄКТ

Фінальний проєкт є підсумковою роботою курсу і передбачає самостійне розгортання повноцінного хмарного додатку з використанням реальної AWS-інфраструктури. Проєкт виконується **індивідуально або в парі**.

---

## Обов'язкові технічні вимоги

Кожен проєкт незалежно від теми повинен містити такі компоненти:

- **EC2** — розгорнутий інстанс, на якому працює додаток.
- **База даних** — DynamoDB або БД, розгорнута безпосередньо на EC2 (PostgreSQL, MySQL, MongoDB тощо).
- **GitHub** — код проєкту зберігається у репозиторії з коректним `.gitignore`, що виключає `.env` та ключі. Деплой на EC2 здійснюється з GitHub — через `git clone` / `git pull` або налаштований GitHub Actions workflow.
- **systemd** — додаток працює у фоновому режимі як systemd-сервіс з увімкненим автозапуском через `systemctl enable`.
- **Змінні середовища** — всі чутливі дані (токени, паролі, endpoint БД) зберігаються у `.env` файлі та підключаються через `EnvironmentFile` у сервісі, а не прописуються у коді.

---

## Вимоги до репозиторію

- `README.md` з описом проєкту, інструкцією з розгортання та переліком використаних AWS-сервісів.
- `requirements.txt` або `package.json` із зафіксованими версіями залежностей.
- Бажано — наявність GitHub Actions workflow для автоматичного деплою при пуші у `main`.

> Робота в парі передбачає вищий рівень складності: більш розвинута функціональність, чіткий розподіл задач між учасниками (через GitHub Issues або коміти від обох акаунтів) та складніша архітектура.

---

## Ідеї для проєктів

Нижче наведено десять ідей, кожна з яких органічно вписується в обов'язковий стек, але при цьому виходить за межі стандартного CRUD-додатку. Ви можете обрати одну з них або запропонувати власну.

---

### 1. 📈 Персональний щоденник настрою з аналітикою

Telegram-бот або веб-інтерфейс, куди користувач щодня вносить короткий запис і оцінку дня від 1 до 10. Система зберігає записи у БД, а на окремій сторінці будує графік настрою за тиждень і місяць, виводить середні показники та найчастіші слова у записах. Незвично тим, що це не утиліта, а особистий інструмент із реальною цінністю для користувача.

**Стек:** EC2 + PostgreSQL на EC2 або DynamoDB + aiogram або Django

---

### 2. 🔔 Трекер цін на товари зі сповіщеннями

Сервіс, який приймає від користувача URL товару з Rozetka або Prom, періодично перевіряє ціну через HTTP-запит, зберігає історію у БД і надсилає сповіщення у Telegram, коли ціна опускається нижче заданого порогу. На EC2 працює фоновий планувальник (APScheduler або cron), що перевіряє ціни кожні кілька годин.

**Стек:** EC2 + PostgreSQL на EC2 або DynamoDB + FastAPI або aiogram

---

### 3. 📄 Генератор резюме через форму

Django або FastAPI застосунок, куди користувач вводить свої дані через форму, а система на основі шаблону генерує відформатоване резюме у PDF, зберігає його в S3 і надсилає посилання на скачування. Опціонально — підключення OpenAI API для автоматичного формулювання опису досвіду.

**Стек:** EC2 + DynamoDB або PostgreSQL на EC2 + Django або FastAPI + S3

---

### 4. 📝 Публічна дошка анонімних зізнань

Веб-додаток у стилі анонімних записок, де користувачі залишають короткі записи без реєстрації. Записи зберігаються у БД з лічильником переглядів і лайків. Нові записи отримують статус "на перевірці" і публікуються лише після автоматичної перевірки на заборонені слова. Проста ідея з нетривіальною логікою станів.

**Стек:** EC2 + PostgreSQL на EC2 або DynamoDB + Django або FastAPI

---

### 5. 🧠 Бот-репетитор з англійської з флеш-картками

Telegram-бот, який зберігає у БД словниковий запас користувача, щодня надсилає йому 5 нових слів з перекладом і через годину перевіряє засвоєння у форматі вікторини. Веде статистику правильних і неправильних відповідей, адаптує складність. Демонструє роботу з розкладом задач і збереженням стану діалогу.

**Стек:** EC2 + DynamoDB або PostgreSQL на EC2 + aiogram

---

### 6. 🔗 Сервіс скорочення посилань з аналітикою

Власний аналог bit.ly. Користувач вводить довге посилання, отримує коротке. При кожному переході система фіксує у БД час, країну (за IP), браузер і реферер. На окремій сторінці — дашборд з графіками переходів по днях і джерелах. Класична ідея з нетривіальною аналітичною частиною.

**Стек:** EC2 + PostgreSQL на EC2 або DynamoDB + Django або FastAPI

---

### 7. 🤝 Колаборативний список справ у реальному часі

Веб-додаток, де кілька користувачів спільно ведуть список задач у реальному часі через WebSocket. Зміни від одного користувача миттєво відображаються в інших вкладках без перезавантаження сторінки. Цікавий тим, що потребує розуміння постійних з'єднань і асинхронності на сервері.

**Стек:** EC2 + PostgreSQL на EC2 + Django Channels або FastAPI WebSocket

---

### 8. 📦 Автоматичний архів Telegram-каналу

Бот підписується на публічний Telegram-канал, парсить нові пости та зберігає їх у БД з тегами і повнотекстовим пошуком. Через простий веб-інтерфейс можна шукати у архіві за ключовими словами і датою. Демонструє роботу із зовнішніми API, фоновими задачами і пошуком у БД.

**Стек:** EC2 + PostgreSQL на EC2 або DynamoDB + aiogram + Django або FastAPI

---

### 9. 🗣️ Сервіс для командних стендапів

Telegram-бот або веб-форма, куди кожен член команди щоранку надсилає три речі: що зробив вчора, що планує сьогодні, чи є блокери. Система збирає відповіді у БД і о певній годині надсилає у спільний Telegram-чат зведений звіт по всій команді. Реальний інструмент із планувальником і агрегацією даних.

**Стек:** EC2 + DynamoDB або PostgreSQL на EC2 + aiogram

---

### 10. 🗒️ Персональний API для нотаток з Markdown

REST API, де авторизований користувач може створювати, редагувати і видаляти нотатки у форматі Markdown. Система зберігає їх у БД, конвертує Markdown у HTML при запиті та дає змогу шукати за тегами. До API додається мінімальний веб-інтерфейс або Telegram-бот як клієнт. Незвично тим, що студент є одночасно розробником API і його споживачем.

**Стек:** EC2 + PostgreSQL на EC2 або DynamoDB + FastAPI
